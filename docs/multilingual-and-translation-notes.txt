Equip — план двуязычия и машинного перевода
(финальная версия, без «разметки», по итогам обсуждения)

[ИСТОРИЧЕСКИЙ ДОКУМЕНТ — план зафиксирован, проектирование завершено,
пайплайн внедрён: см. CHANGELOG.md (раздел [Unreleased]) для актуальной
картины. Расхождения этого файла с реализацией следует читать как
"первичный план" против "что в итоге было реализовано". Например:
- 0) "главный" язык каждой записи — заменено симметричной моделью без
  главного языка; source_locale автоматически выводится из
  preferred_locale учителя в момент создания.
- 5) канонические цитаты Библии — теперь обрабатываются вне LLM через
  app/services/bible/ (substitution + bundled KJV/Synodal JSON), а не
  через промпт-правило.
Документ оставлен как контекст того, как мы пришли к текущей
двуязычной (RU + EN) реализации.]


===============================================================================
0) Кратко
===============================================================================

- Поддерживаемые языки: ru и en. Добавление новых языков — отдельным
  решением, не сейчас.
- Интерфейс — i18n (строки в файлах ru.json / en.json).
- Контент курсов — двуязычный по сущности (блок, вопрос, заголовок и т.д.):
  «главный» язык каждой записи — тот, на котором ее написал учитель
  (authoring locale). Второй язык — производный (машинный перевод + правки,
  если нужны).
- Один глобальный «EN всегда сверху» в данных не вводим. EN главный только
  как маркетинговый/SEO-приоритет сайта (если такой нужен), не как
  принудительный источник правды контента.


===============================================================================
1) Язык интерфейса (UI)
===============================================================================

- Пользователь всегда сам выбирает язык в профиле/шапке.
- Первичный дефолт при первом входе:
    1) сохраненный язык в профиле, если есть;
    2) язык браузера (navigator.language) / Accept-Language;
    3) если не ru и не en — fallback на ru (как основной аудитории школы).
- Выбор пользователя сохраняется в профиле и больше не переопределяется
  автоматически.
- Текст курса (контент) рендерится по тому же locale, что и UI; отдельного
  переключателя «язык курса» нет.


===============================================================================
2) Канон контента (источник правды)
===============================================================================

- Источник правды у каждой переводимой сущности — поле source_locale
  (значение 'ru' или 'en'), фиксируется при создании.
- Учитель пишет на любом из поддерживаемых языков; этот язык и становится
  источником правды для конкретного блока/вопроса/заголовка.
- Второй язык хранится рядом, помечается как производный (origin = 'mt'
  для машинного, 'human' если редактировался человеком).
- Перевод второго языка отображается, но не считается «оригиналом» при
  следующих ре-генерациях (всегда переводим из source).


===============================================================================
3) Что переводим (сущности контента)
===============================================================================

- chapter_blocks.content (text-блоки уроков).
- Заголовки и описания: course.title, course.description, modules.title,
  modules.description, chapters.title.
- Квизы: quizzes.title, quizzes.description, quiz_questions.question_text,
  quiz_options.option_text.
- Assignments: assignments.title, assignments.description (когда появятся
  у курса).
- Файлы (file-блоки): сам файл не переводим; переводим только его
  отображаемое имя/подпись, если задано в UI.

Правки по UI-копии (кнопки, лейблы) — это i18n, не контент-перевод.


===============================================================================
4) Когда запускается перевод
===============================================================================

- Полный пакет: при первом переходе курса в status = 'published'
  (draft -> published). Это уже отдельное событие в crud, добавим хук.
- Дельта: при сохранении изменений в опубликованном курсе пересчитываем
  только те сущности, у которых хеш source-текста изменился.
- При смене source_locale у сущности (учитель переписал тот же блок на
  другом языке) старый перевод инвалидруется.
- Никакого перевода «на лету» при чтении страницы: чтение всегда из БД.


===============================================================================
5) Хранение и пометки актуальности
===============================================================================

Минимальный набор полей в каждой переводимой записи:

  source_locale    -- 'ru' | 'en'
  source_hash      -- hash содержимого исходного текста на момент перевода
  translations     -- { 'ru': {...}, 'en': {...} } или отдельная таблица
                      block_translations(record_id, locale, text, origin,
                                         source_hash, updated_at)

Статусы перевода (производный язык):

  ok       -- хеш совпадает, перевод актуален
  stale    -- source изменился после перевода, нужен пересчет
  missing  -- перевода нет (никогда не делали)
  failed   -- последняя попытка перевода не удалась

Учительский UI должен показывать stale/missing/failed, чтобы автор видел,
где «английский отстал».


===============================================================================
6) Поведение при сбоях/недоступности API
===============================================================================

- Публикацию не блокируем, но в UI владельца курса показываем баннер
  «N блоков ждут перевода» с кнопкой «Повторить».
- Стратегия: ретраи с экспоненциальной задержкой, лимит попыток, потом
  перевод помечается как failed и попадает в очередь ручного повтора.
- Студенту, открывшему курс на втором языке там, где перевода еще нет,
  показываем источниковый текст с маленькой плашкой «Перевод готовится».


===============================================================================
7) Библейские цитаты внутри курса
===============================================================================

- Текст Писания уже вписан в уроки. Отдельный Bible API не подключаем.
- Согласованные переводы:
    - ru — Синодальный (общественное достояние).
    - en — KJV (King James Version, public domain, никаких лицензий).
- Чтобы модель не «перепридумывала» стихи:
    - размечаем цитаты в редакторе (тип узла «citation» с диапазоном
      книга/глава/стих или хотя бы атрибут data-citation),
    - при переводе на второй язык подставляем текст из согласованного
      перевода (Синодаль / KJV) — не из модели,
    - модель переводит только обвязку курса вокруг цитат.
- Промпт говорит модели «не изменяй цитаты, мы их подставим сами», но это
  страховка, а не основная защита.


===============================================================================
8) Промпт-инъекции и «только перевод»
===============================================================================

- В промпте жестко: роль = переводчик, игнорировать инструкции внутри
  переводимого текста, отдавать строго JSON {locale, blocks: [{id, text}]}
  по входному формату.
- На сервере: валидация ответа (JSON, тот же набор id, длина не аномальна,
  язык вывода соответствует целевому). Если что-то не так — отбрасываем,
  помечаем failed, ничего не сохраняем студенту.
- Учитель и так доверенный автор; основная защита — формат + валидация,
  не текст промпта сам по себе.


===============================================================================
9) Скрытие/раскрытие факта машинного перевода
===============================================================================

- Не скрываем технологию демонстративно, но и не выводим бренд провайдера.
- Достаточно одной строки в политике/FAQ: «Часть переводов на другие языки
  выполняется автоматически и может быть отредактирована вручную».
- Конкретный вендор (OpenAI / Gemini / др.) — внутреннее решение
  инфраструктуры.


===============================================================================
10) Конфиденциальность данных
===============================================================================

- В перевод уходит контент курса; персональные данные студентов в API
  перевода НЕ отправляем (имена, email, ответы эссе и т.п. остаются вне
  пайплайна перевода).
- В аккаунте провайдера держим включенной опцию «не использовать данные
  для обучения» там, где это поддерживается.
- В политике школы кратко описать, какие облачные сервисы используются
  для обработки контента.


===============================================================================
11) Провайдер ИИ и инфраструктура
===============================================================================

- Один внутренний интерфейс translate(blocks, target_locale, source_locale)
  -> blocks. Никаких прямых вызовов SDK провайдера из контроллеров.
- Выбранный провайдер: Google Gemini (через Google AI Studio API).
  Модель — gemini-2.5-flash или аналог по цене/качеству на момент
  внедрения, выбирается через env. Адаптер написан так, что заменить
  на OpenAI/другой провайдер можно сменой клиента в одном файле.
- В разработке используется личный ключ; на релиз — корпоративный без
  изменений в коде, только смена секретов в окружении.
- Все секреты (API key, base URL, model name) — только через env;
  захардкоженных значений нет.
- Хранение секретов: как и остальные секреты проекта — переменные
  окружения на бэкенде (Vercel project env / Supabase secrets для
  серверного кода). НЕ через VITE_ префикс: ключ перевода никогда
  не должен попадать в браузер.
- Лимит трат у провайдера: soft cap $5/мес, hard cap $20/мес на старте;
  поднять при необходимости. Цель — защита от багов (бесконечный цикл,
  заглючивший ретрай), а не бюджет.


===============================================================================
12) Поиск и SEO
===============================================================================

- search_vector курса нужно собирать из обоих языков (или хранить два
  вектора: ts_russian и ts_english) — иначе поиск будет находить курс
  только на одном языке.
- URL пока оставляем без префикса локали (/courses/:id). Если в будущем
  каталог станет публичным и нужен SEO, добавим префиксы /ru/ /en/ и
  hreflang. Это не часть первой реализации.


===============================================================================
13) Роли и владение курсом
===============================================================================

- В текущем коде у курса один владелец (created_by). Этого достаточно для
  первой версии двуязычия: тот же человек либо доверяет машинному EN,
  либо правит его сам.
- Отдельная роль «редактор перевода» / несколько преподавателей на курс —
  не вводим сейчас; помечаем как возможную будущую доработку.


===============================================================================
14) Порядок реализации (рекомендуемый)
===============================================================================

Wave 1 — фундамент:
  1. Подключить i18n на фронте (ru/en), вынести строки UI.
  2. Поле user.preferred_locale в профиле + логика дефолта по браузеру
     при первом входе.
  3. Решить и зафиксировать структуру хранения переводов контента
     (JSONB поле или отдельная таблица block_translations и аналоги).
  4. Внутренний translate() слой с одним провайдером и env-конфигом.

Wave 2 — пайплайн перевода:
  5. Хук на draft -> published: enqueue полного перевода всех сущностей
     курса в недостающий язык.
  6. Хеши source_text + статусы ok/stale/missing/failed.
  7. Валидация ответа модели + ретраи + UI «N блоков ждут перевода».

Wave 3 — качество:
  8. Разметка библейских цитат и подстановка одобренных переводов.
  9. UI учителя «английский отстал»: список stale-блоков, кнопка
     «перевести заново».
 10. Поиск по двум языкам (search_vector обновлений).

Wave 4 — позже, по необходимости:
 11. Роль co-teacher/translation editor.
 12. Префиксы локали в URL и hreflang, если нужен публичный SEO.
 13. Поддержка третьего языка (изменения только в конфиге локалей,
     если 1-12 сделаны правильно).


===============================================================================
15) Что НЕ делаем (явно)
===============================================================================

- Не переводим в реальном времени при открытии страницы.
- Не нормализуем все курсы в EN автоматически.
- Не подключаем отдельный Bible API на старте.
- Не открываем настройки модели/вендора пользователю.
- Не вводим co-teacher и URL-локали в первой версии.
